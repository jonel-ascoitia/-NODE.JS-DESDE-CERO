🟢 DOCUMENTO DE CIERRE — DÍA 5
ESTUDIANTE: Esteban

1. Explica el flujo completo desde que un admin hace login hasta que puede agregar un plato.
Respuesta: El admin envía datos a `/auth/login`. El controlador llama a `authService.login()` y éste firma un JWT. Para agregar un plato, el admin envía el token en el header `Authorization`. `middlewares/verifyToken.js` valida el token y da paso al controlador del menú.

2. ¿Por qué bcrypt.compare() puede verificar la contraseña si el hash es irreversible?
Respuesta: Porque es un proceso de "una sola vía". La función vuelve a hashear la contraseña del login y si el resultado es igual al de la DB, sabemos que la clave es correcta sin necesidad de descifrarla.

3. Si alguien roba un token JWT válido, ¿qué puede hacer con él? ¿Qué limita el daño? ¿Qué agregarías?
Respuesta: El atacante puede actuar como admin durante la vida útil del token. Lo acota el parámetro `expiresIn: '24h'`. Para dar más seguridad, usaría tokens que expiren en 1 hora o rotación de los mismos.

4. ¿Qué importancia tiene process.env.JWT_SECRET?
Respuesta: Sin la clave, el servidor no puede verificar si el token es legítimo. Es vital tenerla fuera del código fuente (`.env`) para que nadie genere tokens falsos por fuera.
