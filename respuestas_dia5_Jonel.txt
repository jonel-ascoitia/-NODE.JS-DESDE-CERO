ESTUDIANTE: Jonel

1. Explica el flujo completo desde que un admin hace login hasta que puede agregar un plato.
Respuesta: El admin entra a `/auth/login`, se valida su password en `auth.service.js` y recibe un JWT. Luego, envía ese token en el header `Authorization` para peticiones `/menu`. El archivo `verifyToken.js` valida la firma y deja pasar la petición al controlador del menú.

2. ¿Por qué bcrypt.compare() puede verificar la contraseña si el hash es irreversible?
Respuesta: No la descifra. Lo que hace es hashear la clave que escribes en el login y comparar si ese nuevo hash es idéntico al que está guardado en MongoDB. Si coinciden, la clave es correcta.

3. Si alguien roba un token JWT válido, ¿qué puede hacer con él? ¿Qué limita el daño? ¿Qué agregarías?
Respuesta: Puede suplantar al admin hasta que el token expire. Lo que limita el daño es que el token solo dura 24 horas. Para mejorarlo, usaría HTTPS siempre y tokens de acceso más cortos (ej. 1 hora).

4. ¿Qué importancia tiene process.env.JWT_SECRET?
Respuesta: Es la llave maestra. Sin ella, nadie puede crear o validar tokens. Se guarda en `.env` para que no esté expuesta en el código de GitHub.
